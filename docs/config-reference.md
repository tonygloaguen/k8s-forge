# app.yaml Configuration Reference

## Introduction

`app.yaml` is the source of truth for `k8s-forge`. The tool reads this file,
validates it, and uses it to render Kubernetes manifests for a stateless
containerized web application.

Every Kubernetes value generated by `k8s-forge` comes from `app.yaml`: names,
namespace, image, ports, replicas, environment variables, secrets, resource
settings, and probe paths. The implementation must stay generic and must not
hardcode an application-specific name.

## Minimal Example

```yaml
app:
  name: demo-app
  namespace: demo
  image: demo-app:latest
  containerPort: 8000
  replicas: 1

service:
  enabled: true
  port: 80
```

## Complete Example

```yaml
app:
  name: demo-app
  namespace: demo
  image: demo-app:1.0.0
  containerPort: 8000
  replicas: 2

config:
  APP_ENV: "dev"
  LOG_LEVEL: "info"

secrets:
  API_TOKEN: "change-me"

service:
  enabled: true
  port: 80

resources:
  requests:
    cpu: "50m"
    memory: "64Mi"
  limits:
    cpu: "250m"
    memory: "128Mi"

probes:
  liveness: "/healthz"
  readiness: "/readyz"

ingress:
  enabled: false
  host: null
```

## Field Reference

| Field | Type | Required | Default via `init` | Constraints | Kubernetes usage |
| --- | --- | --- | --- | --- | --- |
| `app.name` | string | Yes | command argument | non-empty | Deployment name, Service name, labels, ConfigMap/Secret name prefix |
| `app.namespace` | string | Yes | `app.name` | non-empty | Namespace name and resource namespace |
| `app.image` | string | Yes | `<app-name>:latest` | non-empty | Deployment container image |
| `app.containerPort` | integer | Yes | `8000` | `1` to `65535` | Deployment container port and probe port; Service `targetPort` |
| `app.replicas` | integer | Yes | `1` | greater than or equal to `1` | Deployment replica count |
| `config` | map of strings | No | `APP_ENV`, `LOG_LEVEL` placeholders | keys and values must be strings | ConfigMap `data`, injected with `envFrom` when non-empty |
| `secrets` | map of strings | No | `API_TOKEN: "change-me"` | keys and values must be strings | Secret `stringData`, injected with `envFrom` when non-empty |
| `service.enabled` | boolean | No | `true` | boolean only | Controls whether a Service is rendered |
| `service.port` | integer | No | `80` | `1` to `65535` | Service exposed port |
| `resources.requests.cpu` | string | No | `50m` | optional string | Deployment container resource requests |
| `resources.requests.memory` | string | No | `64Mi` | optional string | Deployment container resource requests |
| `resources.limits.cpu` | string | No | `250m` | optional string | Deployment container resource limits |
| `resources.limits.memory` | string | No | `128Mi` | optional string | Deployment container resource limits |
| `probes.liveness` | string | No | `/healthz` | optional string | Deployment HTTP liveness probe path |
| `probes.readiness` | string | No | `/readyz` | optional string | Deployment HTTP readiness probe path |
| `ingress.enabled` | boolean | No | `false` | boolean only | Validated only; not rendered in the MVP |
| `ingress.host` | string or null | No | `null` | optional string or null | Reserved for future Ingress support |

## Section Details

### `app`

The `app` section defines the main identity and runtime settings of the
application.

`name` is used for the Deployment, Service, common labels, and derived names
such as `<app.name>-config` and `<app.name>-secret`. `namespace` isolates the
resources in Kubernetes. `image` is the container image used by the Deployment.
`containerPort` is the port exposed by the container. `replicas` is the desired
number of Pods.

### `config`

The `config` section contains non-sensitive environment configuration. When it
is non-empty, `k8s-forge` renders a ConfigMap and injects it into the container
with `envFrom`.

Do not put secrets in `config`.

### `secrets`

The `secrets` section contains placeholder secret values for the MVP. When it is
non-empty, `k8s-forge` renders a Kubernetes Secret and injects it into the
container with `envFrom`.

The MVP uses `stringData` for readability. This is educational and convenient
for local examples, but real secrets must not be committed to source control.

### `service`

The `service` section controls Service generation. A Service is rendered only
when `service.enabled` is `true`.

`service.port` is the port exposed by the Service. The Service `targetPort` is
always taken from `app.containerPort` so the Service routes to the container
port declared in the same `app.yaml`.

### `resources`

The `resources` section defines optional container requests and limits. Setting
requests and limits is recommended for Kubernetes workloads because it helps the
scheduler place Pods and helps enforce resource boundaries.

Example:

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"
```

### `probes`

The `probes` section defines optional HTTP liveness and readiness probe paths.
When set, the Deployment uses `app.containerPort` as the HTTP probe port.

These endpoints must actually exist in the application. A missing or incorrect
probe endpoint can cause Kubernetes to restart Pods or keep them out of Service
endpoints.

### `ingress`

The `ingress` section is validated but not rendered in the MVP. It exists to
reserve a stable configuration shape for a future Ingress feature.

Setting `ingress.enabled: true` does not currently generate an Ingress manifest.

## Invalid Examples

### Port set to `0`

```yaml
app:
  name: demo-app
  namespace: demo
  image: demo-app:latest
  containerPort: 0
  replicas: 1
```

`app.containerPort` must be between `1` and `65535`.

### Replicas set to `0`

```yaml
app:
  name: demo-app
  namespace: demo
  image: demo-app:latest
  containerPort: 8000
  replicas: 0
```

`app.replicas` must be greater than or equal to `1`.

### Image missing

```yaml
app:
  name: demo-app
  namespace: demo
  containerPort: 8000
  replicas: 1
```

`app.image` is required because the Deployment needs a container image.

### Required application name missing

```yaml
app:
  namespace: demo
  image: demo-app:latest
  containerPort: 8000
  replicas: 1
```

`app.name` is required because it drives resource names, labels, and selectors.

## Best Practices

- Use an explicit image tag for real projects instead of `latest`.
- Adapt `resources` to the expected workload before applying manifests.
- Never commit real secrets in `app.yaml` or generated manifests.
- Run `k8s-forge check app.yaml` before rendering.
- Run `dry-run` and `diff` before `apply`.

## Associated Commands

```bash
k8s-forge init demo-app
k8s-forge check app.yaml
k8s-forge render app.yaml --output generated/
k8s-forge dry-run app.yaml --output generated/
k8s-forge diff app.yaml --output generated/
k8s-forge apply app.yaml --output generated/
```
